# Solution — G5.2, G4.1, G4.4: RMW serialization + atomic write for session-history sidecar

> Scope: design only. No code modified.
> Cites: LD2 (lease-lifecycle writes only), LD3 (separate sidecar), LD4 (UPSERT by sessionId),
> LD5 (owner excluded from array).
> Status: ready for proposal / implementation review.

## Recap — what each gotcha demands

| Gotcha | Demand |
|---|---|
| G5.2 | Atomic write + fsync so a torn write / crash cannot corrupt the history file. |
| G4.1 | Serialize the read-modify-write (RMW) so reaper and acquirer cannot lose updates to `sessions`. |
| G4.4 | Prevent the "stale snapshot overwrites fresh" regression — a symptom of G4.1. |

All three collapse to one architectural move + one defensive rule, below.

---

## G4.1 — Serialization: piggyback on the existing lease mutex (recommended)

### Options evaluated

| Option | Mechanism | Verdict |
|---|---|---|
| A. `flock(2)` advisory lock | `fs.open(..., 'wx')` + `flock(fd, LOCK_EX)` around every RMW | Heavy. Introduces a second lock primitive alongside the lease. Node's `fs` has no native flock — needs `fs-ext` dep or raw binding. Cross-platform pain. |
| B. mkdir-based lockfile | `mkdir(.locks/.hist-<goalId>.lock)` (atomic on POSIX), write, `rmdir` | Adds a third file per goal (lease + sidecar + lockfile). Stale-lockfile cleanup hazard on crash. |
| C. O_EXCL lockfile | `fs.openSync(path, 'wx')` | Same as B, different primitive. Same objections. |
| D. CAS via content hash + rename | Read file + hash → write tmp → rename only if on-disk hash unchanged (impossible to make atomic in pure Node; needs `linkat`/`renameat2`). | Not implementable portably in Node. Even Linux `renameat2(RENAME_EXCHANGE)` is out of reach. |
| **E. Piggyback on the lease mutex** | Only the current lease holder may write the sidecar. No new primitive. | ✅ **Recommended.** |

### Why E wins

The lease mutex already establishes a single writer (LD2's "only at lifecycle points" invariant).
Extending that invariant to the sidecar is free:

- `acquireLock` is the *only* path that transitions ownership. By the time a session holds the lease,
  the previous holder is either (a) dead (stale → reaped → file gone) or (b) cleanly released.
- `refreshLease` is the *only* path that touches the lease while held, and it is identity-checked
  (`existing.owner.sessionId !== self.sessionId` → `lostLock: true`).
- `releaseLock` is the *only* path that surrenders ownership, and it TOCTOU-re-reads before unlink.

**Critical insight:** the reaper (`reapStaleLock` / `reapOrphanedLocks`) only deletes the **lease**
`.lock` file. It never touches the sidecar. So there is no concurrent writer from the reaper against
the sidecar — the reaper's blast radius is bounded to the lease, exactly as it was before LD3.

The only concurrency the sidecar sees is:
1. Old holder A writing `lastWorkedAt` on `refreshLease` heartbeat, while
2. New holder B (having reaped A's stale lease) reading A's sidecar on `acquireLock`.

But (1) and (2) cannot overlap: B's `acquireLock` only proceeds after A's lease is *stale*
(PID dead or expired). A dead/expired A is not writing heartbeats. **The lease lifecycle already
serializes the sidecar writers.** No new primitive needed.

### The invariant (formal)

> **If you do not hold the lease, you do not write the sidecar.**

Enforced at the write site (covers G3.1 too — out of this worker's scope but the guard lives here):

```ts
// Pseudocode — matches existing style of extensions/goal-lock.ts
function upsertSessionEntry(
  cwd: string,
  goalId: string,
  self: LockOwner,
  lease: GoalFocusLock,  // MUST be the currently-held lease for goalId
): void {
  // GUARD: caller proves lease ownership. This is the G4.1 + G3.1 enforcement point.
  if (lease.owner.sessionId !== self.sessionId) {
    logGoalTrace(cwd, {
      level: "warn",
      step: "history.upsertSessionEntry.notOwner",
      goalId,
      message: `rejected sidecar write: lease owned by ${lease.owner.sessionId}, not ${self.sessionId}`,
    });
    return; // fail-open: history is advisory, never crash the host
  }
  const sessions = readSessions(cwd, goalId); // G2.3: normalizes missing → []
  const updated = upsertBySessionId(sessions, self.sessionId, { /* see LD4 */ });
  writeSessionsAtomic(cwd, goalId, updated);
}
```

The lease object is passed in, not re-read — the caller (`acquireLock` / `refreshLease` /
`releaseLock`) already has the authoritative lease in hand and has already passed its own
identity check.

## G4.4 — Spread-miss regression: killed by the "read fresh → mutate → write" rule

### The hazard (from gotcha appendix)

```ts
// WRONG — constructs updated from a stale snapshot captured earlier:
const updated = { ...staleLeaseSnapshot, expiresAt, heartbeatAt };
writeSessionsAtomic(cwd, goalId, updated.sessions); // ← rewrites an OLD sessions array
```

### The fix: two-part rule

**Rule 1 (architectural):** `sessions` is NEVER constructed from a snapshot. It is read fresh from
the sidecar *inside* the RMW function, mutated, and written back — all in one synchronous call.

```ts
function upsertSessionEntry(cwd, goalId, self, lease): void {
  // ... guard ...
  const fresh = readSessions(cwd, goalId);        // READ fresh, right here, right now
  const mutated = upsertBySessionId(fresh, self.sessionId, entry);
  writeSessionsAtomic(cwd, goalId, mutated);      // WRITE what we just read
}
```

There is no path where a stale `sessions` array flows from a lease snapshot into a sidecar write.
The two stores are decoupled by construction (LD3's whole point).

**Rule 2 (defensive, type-level):** the sidecar write function takes `SessionEntry[]`, NOT
`GoalFocusLock` or `Partial<...>`. It cannot "spread" a lease and accidentally drop or regress
`sessions`:

```ts
// Signature forbids spread-miss by construction
function writeSessionsAtomic(
  cwd: string,
  goalId: string,
  sessions: SessionEntry[],   // ← whole array, not Partial<HistoryFile>
): void { ... }
```

A caller cannot do `{ ...lock, sessions }` through this function — the function doesn't accept
`lock`. G4.4 is structurally impossible.

### How G4.1 prevents the interleaving variant

The interleaving variant of G4.4 (read at T0, someone writes at T1, we write at T2 → lost T1's
update) is prevented by G4.1's invariant: the someone-at-T1 must hold the lease, but *we* also
hold the lease (else we wouldn't be in `upsertSessionEntry`). Two holders of the same lease is a
contradiction. So T1 cannot happen during our RMW window.

## G5.2 — Atomic write + fsync strategy

### The existing pattern (lease file)

`writeLockAtomic` already does tmp + rename:
```ts
const tmp = path.join(dir, `.${lock.owner.pid}.${Date.now()}.tmp`);
fs.writeFileSync(tmp, JSON.stringify(lock));
fs.renameSync(tmp, final);
```

This is *rename-atomic* (POSIX guarantees the rename is atomic) but **not crash-durable**:
`writeFileSync` may return before data hits disk (kernel page cache), and `renameSync` may
journal the directory entry before the data block is durable. A power loss between the write and
a true fsync can leave the renamed file as a zero-length or torn artifact.

For the lease file, this is acceptable — a torn lease is reaped on sight, self-healing. For the
history sidecar, LD3 was chosen *precisely* because history is meant to survive crashes. So the
sidecar must be crash-durable.

### Strategy: tmp → fsync(tmp) → rename → fsync(dir)

```ts
function writeSessionsAtomic(
  cwd: string,
  goalId: string,
  sessions: SessionEntry[],
): void {
  ensureSessionsDir(cwd);
  const dir = lockDir(cwd);              // same .locks/ dir as the lease
  const final = sessionsPath(cwd, goalId);
  const tmp = path.join(dir, `.sessions.${process.pid}.${Date.now()}.tmp`);

  const payload: SessionsFile = {
    version: 1,                          // OT6: schema version
    goalId,
    sessions,                            // whole array, LD4 upsert result
  };

  try {
    // 1. Write tmp file fully.
    const fd = fs.openSync(tmp, "w");
    try {
      fs.writeFileSync(fd, JSON.stringify(payload));
      fs.fsyncSync(fd);                  // ← NEW: durable data before rename
    } finally {
      fs.closeSync(fd);
    }
    // 2. Atomic rename.
    fs.renameSync(tmp, final);
    // 3. Durability of the directory entry (the rename itself).
    fsyncDirSync(dir);                   // ← NEW: see below
  } catch (err) {
    logGoalTrace(cwd, {
      level: "warn",
      step: "history.writeSessionsAtomic",
      goalId,
      message: `failed to write sessions sidecar ${final}`,
      error: previewError(err),
    });
    try { fs.unlinkSync(tmp); } catch { /* may not exist */ }
    // FAIL-OPEN: history is advisory. Never throw.
  }
}

function fsyncDirSync(dir: string): void {
  let dh: number | undefined;
  try {
    dh = fs.openSync(dir, "r");
    fs.fsyncSync(dh);                    // POSIX: fsync on a dir fd persists its entries
  } catch (err) {
    logGoalTrace(/* cwd passed in or captured */, {
      level: "warn",
      step: "history.fsyncDir",
      message: `directory fsync failed for ${dir} (history durability reduced)`,
      error: previewError(err),
    });
    // FAIL-OPEN: we logged; do not throw. Worst case: rename not yet durable on crash.
  } finally {
    if (dh !== undefined) {
      try { fs.closeSync(dh); } catch { /* ignore */ }
    }
  }
}
```

### Ordering guarantees

```
  write(tmp) ──▶ fsync(tmp) ──▶ rename(tmp→final) ──▶ fsync(dir)
       │              │                  │                 │
       │              │                  │                 └─ persists the rename entry
       │              │                  └─ atomic swap (POSIX guarantee)
       │              └─ data block durable BEFORE rename → no torn-final possible
       └─ buffered in page cache
```

Crash points and outcomes:

| Crash at | On-disk state after recovery | History safe? |
|---|---|---|
| Before `write` | old `final` (or none) | ✅ yes (pre-write state) |
| After `write`, before `fsync(tmp)` | old `final`; tmp may be partial | ✅ yes (tmp discarded on retry; `final` untouched) |
| After `fsync(tmp)`, before `rename` | old `final`; complete tmp orphan | ✅ yes (orphan tmp cleaned on next write) |
| After `rename`, before `fsync(dir)` | `final` = new content (durable data); rename *probably* journaled | ✅ almost always (rename is atomic; only risk is rename-entry-loss on power loss, which leaves old `final`) |
| After `fsync(dir)` | `final` = new content, fully durable | ✅ yes |

**Key property:** at no crash point does `final` exist in a *torn* state. Either the old content
or the new (fully-fsync'd) content is visible. This is what G5.2 demanded.

### Fail-open semantics on fsync error

- `fs.fsyncSync` throws on platforms that don't support it, on read-only FS, on certain network
  filesystems, or on permission errors.
- We **log and continue**, never throw. Rationale: history is advisory (LD-spirit); crashing the
  host (e.g., killing a `refreshLease` heartbeat) is strictly worse than a slightly-less-durable
  history entry.
- The rename still happens even if the data fsync failed — better to have a consistent (if not
  crash-durable) file than a dangling tmp.
- Trace logs carry the failure so an operator can detect a degraded durability mode (relevant for
  the OT5 hardening follow-up).

### Platform notes

- **Linux:** `fs.fsyncSync` on a directory fd is supported and persists dirent changes. Standard.
- **macOS:** same — `fsync` on a dir fd flushes the directory.
- **Windows / others:** `fsync` on a directory fd may throw `EINVAL`. The `fsyncDirSync` wrapper
  catches this and logs; the file fsync still applies. Reduced durability on exotic platforms is
  acceptable (the existing codebase is Linux/macOS-first — see `getProcessStartTime`).
- **Network filesystems (NFS, etc.):** fsync semantics are weaker. Same fail-open path applies.

## New file / API surface

### Files

| Path | Purpose | New? |
|---|---|---|
| `.locks/<goalId>.sessions.json` | session history sidecar (LD3) | ✅ new |
| `.locks/.sessions.<pid>.<ts>.tmp` | atomic-write tmp file (G5.2) | ✅ new (ephemeral) |

### API surface (to live in `goal-lock.ts` or a new `goal-history.ts`)

```ts
// Types
export interface SessionEntry {
  sessionId: string;
  lastWorkedAt: string;        // ISO; see G2.2 worker's scope for wall-clock vs monotonic
  // Optional enrichment (G2.1 worker's scope): pid, startTimeMs if sessionId not UUID-grade
}

export interface SessionsFile {
  version: 1;
  goalId: string;
  sessions: SessionEntry[];
}

// Path helpers
export function sessionsPath(cwd: string, goalId: string): string;

// Read (normalizes missing → [], validates shape, OT6 version-aware)
export function readSessions(cwd: string, goalId: string): SessionEntry[];

// Write (atomic + fsync, G5.2; fail-open)
export function writeSessionsAtomic(
  cwd: string,
  goalId: string,
  sessions: SessionEntry[],
): void;

// Upsert primitive (LD4)
export function upsertBySessionId(
  sessions: SessionEntry[],
  sessionId: string,
  patch: { lastWorkedAt: string },
): SessionEntry[];

// Guarded RMW entry point (G4.1 + G3.1 enforcement)
export function upsertSessionEntry(
  cwd: string,
  goalId: string,
  self: LockOwner,
  lease: GoalFocusLock,
): void;
```

### Integration into existing lease lifecycle (LD2)

| Existing fn | New hook | When |
|---|---|---|
| `acquireLock` | After successful verify, before return: `upsertSessionEntry(cwd, goalId, self, verified)` | Stamps the new holder into history. `verified` proves we hold the lease. |
| `refreshLease` | After successful write: `upsertSessionEntry(cwd, goalId, self, updated)` | Bumps `lastWorkedAt` on heartbeat (60s cadence). |
| `releaseLock` | Before `unlinkSync(lockPath)`: `upsertSessionEntry(cwd, goalId, self, existing)` | Final stamp; LD5 means owner is *added* here (excluded while active, included on release). |

**LD5 nuance:** owner is excluded from the array *while active*. On release, the owner transitions
from active → historical, so `releaseLock`'s `upsertSessionEntry` is what first adds the now-former
owner. While active, the lease's `owner` field is canonical (G4.3 mitigation, that worker's scope).

### Reaper interaction (does NOT touch sidecar)

- `reapStaleLock(cwd, goalId)` — unchanged. Deletes `.lock` only.
- `reapOrphanedLocks(cwd, activeGoalIds)` — **enhancement needed (separate concern, devil's-advocate worker flagged).**
  Currently scans `.locks/*.lock`. Should also scan `.locks/*.sessions.json` and reap sidecars
  whose goalId is not active. This is NOT in this worker's scope (it's an orphan-cleanup concern,
  not a durability/RMW concern) but flagged here so it isn't lost.

## Summary table — gotcha → resolution

| Gotcha | Resolution | Mechanism |
|---|---|---|
| **G5.2** | tmp → fsync(tmp) → rename → fsync(dir). `final` is never torn at any crash point. | `writeSessionsAtomic` + `fsyncDirSync`, fail-open on fsync error. |
| **G4.1** | Piggyback on lease mutex: only the holder writes the sidecar. Reaper never touches sidecar. Lease lifecycle already serializes writers. | `upsertSessionEntry` guard: `lease.owner.sessionId !== self.sessionId` → no-op. |
| **G4.4** | (a) `sessions` read fresh inside the RMW function, never from a snapshot; (b) write fn signature takes `SessionEntry[]`, not `Partial<...>` — spread-miss is structurally impossible. | `readSessions → upsertBySessionId → writeSessionsAtomic` in one synchronous call. |
