## G3.1 — "Lease-holder semantics only" is unenforceable at the write site

### Problem

LD2 states the history array mutates *only* at acquireLock/refreshLease/releaseLock. But "only the holder writes" is a **design rule, not a runtime guard**. Any caller — a buggy code path, a future feature, or the existing takeover path — could invoke a history write without holding the lease, silently polluting the array.

### Evidence the hazard is real (not theoretical)

The codebase **already has a non-holder write path** for the lease itself:

```ts
// extensions/goal.ts:2029 (takeover confirmation)
releaseLock(ctx.cwd, goalId);   // ← NO `self` arg → unconditional delete
```

This is the `/goal-focus` takeover: session B steals a lock held by A by calling `releaseLock` *without* proving ownership. The forced release is intentional (user confirmed). But it proves non-holder mutations are an existing pattern — the history API must defend against the same, not assume callers always hold.

### Design — guard at the WRITE site

Put the ownership check **inside** the history-mutation function, not in callers. Defense-in-depth: every entry point is safe regardless of caller discipline.

```ts
// extensions/goal-lock.ts (new sidecar API)

export type SessionHistoryEntry = {
  sessionId: string;        // UUID (see G2.1)
  lastWorkedAt: string;     // ISO 8601 wall-clock (see G2.2)
};

export type SessionHistoryResult =
  | { ok: true }
  | { ok: false; reason: "not-owner" | "no-lease" | "fs-error" };

/**
 * Upsert a session's lastWorkedAt into the history sidecar.
 *
 * GUARD (G3.1): verifies `self` currently holds a LIVE lease for goalId
 * before writing. If not the holder → silent no-op + warn trace. History
 * is advisory metadata, not a security boundary — same FAIL-OPEN posture
 * as the lease lock itself (see file header).
 *
 * Respects: LD2 (lifecycle-only), LD3 (sidecar), LD4 (upsert), LD5 (owner excluded).
 */
export function upsertSessionHistory(
  cwd: string,
  goalId: string,
  self: LockOwner,
  now: number = Date.now(),
): SessionHistoryResult {
  // 1. Ownership guard — the write site check.
  const lock = readLock(cwd, goalId);
  if (!lock) return { ok: false, reason: "no-lease" };
  if (lock.owner.sessionId !== self.sessionId || !isLockHeld(lock)) {
    logGoalTrace(cwd, {
      level: "warn",
      step: "history.upsert.not-owner",
      goalId,
      message: `history write rejected: caller ${self.sessionId} does not hold live lease (owner=${lock.owner.sessionId})`,
    });
    return { ok: false, reason: "not-owner" };
  }

  // 2. Read current sidecar (normalize undefined → [], see G2.3).
  const history = readSessionHistory(cwd, goalId); // returns [] on missing/corrupt

  // 3. UPSERT by sessionId (LD4). Owner excluded (LD5) — the holder is NOT
  //    added here; only released sessions live in the array. The current
  //    holder's "lastWorked" is the live `lock.heartbeatAt`.
  const filtered = history.filter((e) => e.sessionId !== self.sessionId);
  // (no-op if self not present — correct: holder stays out until release)

  try {
    writeSessionHistoryAtomic(cwd, goalId, filtered);
    return { ok: true };
  } catch (err) {
    logGoalTrace(cwd, { level: "warn", step: "history.upsert.write_failed", goalId, error: previewError(err) });
    return { ok: false, reason: "fs-error" };
  }
}
```

### Fail-open vs fail-safe — decision: **FAIL-OPEN**

| Option | Behavior | Verdict |
|---|---|---|
| Silent no-op + warn trace | rejected write logged, no crash | ✅ CHOSEN |
| Throw | crashes heartbeat timer (line 894 `setInterval`) | ❌ breaks lease lifecycle |
| Fail-loud (ctx.ui.notify) | surfaces to user on every heartbeat | ❌ noise; advisory data |

**Rationale**: The file header already declares "All fs operations are FAIL-OPEN: locking is an optimization, not a security boundary." History is the same class of advisory metadata. A rejected write degrades to "this session not recorded" — the lease and all other sessions' history are unaffected. No throw, no crash.

### Where the check lives

Single function `upsertSessionHistory` in `extensions/goal-lock.ts`. Called from:
- `refreshLease` (heartbeat path) — after the existing owner check succeeds
- `releaseLock` (when `self` provided) — before unlink, stamps self into array as the just-released session
- `acquireLock` — does NOT write history on acquire (owner excluded per LD5); only the *previous* holder's final state is already in the array from their releaseLock

**Callers do not re-check ownership** — the guard is canonical. This prevents the "caller forgot to verify" class of bug.

## G2.1 — Session-ID reuse collides in dedup

### Finding: **resolved by existing code — no action needed**

```ts
// extensions/goal.ts:253
const SELF_SESSION_ID = crypto.randomUUID();
```

`crypto.randomUUID()` returns a **RFC 4122 v4 UUID** — 122 bits of cryptographic randomness. Collision probability is negligible (~10⁻³⁷ per pair). Globally unique across process restarts (regenerated each session).

### Implication for LD4 (UPSERT by sessionId)

Keying the history array on `sessionId` alone is **safe**. No need for the `{sessionId, pid, startTimeMs}` tuple fallback proposed in the gotcha. The upsert filter `e.sessionId === self.sessionId` correctly dedups across a single session's acquire/release/re-acquire cycle.

### Hard dependency (must document)

The upsert keying assumes sessionId remains UUID-grade. **If `SELF_SESSION_ID` generation ever changes** (e.g., to a derived/predictable value like `pid@hostname`), the dedup breaks silently. Mitigation:

```ts
// Add a one-time assertion at module load (extensions/goal.ts)
// Guards against future regression in sessionId generation.
const UUID_V4_RE = /^[0-9a-f]{8}-[0-9a-f]{4}-4[0-9a-f]{3}-[89ab][0-9a-f]{3}-[0-9a-f]{12}$/i;
if (!UUID_V4_RE.test(SELF_SESSION_ID)) {
  logGoalTrace(cachedCwd ?? process.cwd(), {
    level: "warn",
    step: "session.id.not-uuid",
    message: `SELF_SESSION_ID is not a UUID v4 — history dedup may collide`,
  });
}
```

This is a **defensive check, not a blocker**. Current code satisfies it.

## G2.2 — Clock skew / non-monotonic time breaks lastWorkedAt ordering

### Decision: **wall-clock ISO 8601 (matches existing fields)**

`lastWorkedAt: string` — ISO 8601 timestamp, same format as `acquiredAt`, `expiresAt`, `heartbeatAt` in `GoalFocusLock`. Generated via `new Date(now).toISOString()`.

### Tradeoff analysis

| Option | Cross-process? | Human-readable? | Monotonic? | Verdict |
|---|---|---|---|---|
| Wall-clock ISO (chosen) | ✅ yes | ✅ "10m ago" | ❌ NTP can regress | ✅ |
| Monotonic (`process.hrtime`) | ❌ per-process only | ❌ raw ns | ✅ | ❌ useless across sessions |
| Hybrid (epoch ms + monotonic seq) | ✅ | ❌ | partial | ❌ over-engineered |

### Why wall-clock is good enough for the resume use case

1. **Single-host personal agent** — the dominant deployment. All sessions on one machine share one clock. Cross-host skew is a non-issue.
2. **Resume use case is advisory display** — "B worked on this 10m ago" is a hint, not an audit log. A transient NTP step-backward causing a 1-second ordering flip is invisible to the user.
3. **Consistency** — matching `heartbeatAt` format means the current holder's liveness and historical sessions' lastWorked use the *same clock domain*. No mixed-format comparison bugs.

### Documented edge (accepted risk)

NTP step-backward: if the clock jumps backward between two `refreshLease` calls within one session, `lastWorkedAt` could decrease. Effect: "most recent session" ordering transiently wrong for the affected session. Acceptable for advisory metadata. **No mitigation** — matching the existing lease fields' posture (they also use wall-clock and are equally affected, by design).

### Ordering contract (explicit)

- History array is **NOT sorted by lastWorkedAt**. It is append/upsert-order (insertion time).
- Consumers that want "most recent first" MUST sort on read: `history.sort((a,b) => Date.parse(b.lastWorkedAt) - Date.parse(a.lastWorkedAt))`.
- This prevents array-order from masquerading as recency-order and hides clock skew behind an explicit consumer-side sort.

## G2.3 — `sessions?` optional forces null-check everywhere

### Decision: **normalize to `[]` at read boundary; internal code never sees `undefined`**

Mirror the `readLockDetailed` discriminated-return pattern. The sidecar reader is the single point where `undefined`/missing/corrupt collapses to `[]`.

```ts
// extensions/goal-lock.ts (new sidecar API)

export interface SessionHistoryFile {
  goalId: string;
  version: 1;                    // schema version (G3.4 mitigation, OT6)
  sessions: SessionHistoryEntry[]; // ALWAYS present after read; never undefined
}

export type ReadSessionHistoryResult =
  | { status: "found"; file: SessionHistoryFile }
  | { status: "missing" }        // ENOENT
  | { status: "error" };         // EACCES, corrupt JSON, invalid shape

export function readSessionHistoryDetailed(
  cwd: string,
  goalId: string,
): ReadSessionHistoryResult {
  try {
    const data = fs.readFileSync(sessionsPath(cwd, goalId), "utf8");
    let parsed: unknown;
    try {
      parsed = JSON.parse(data);
    } catch {
      return { status: "error" };  // corrupt JSON → error (not missing)
    }
    if (!isValidSessionHistoryShape(parsed)) return { status: "error" };

    const file = parsed as SessionHistoryFile;
    // NORMALIZE (G2.3): missing/undefined sessions → [].
    // Legacy or hand-edited files without the field are tolerated.
    if (!Array.isArray(file.sessions)) file.sessions = [];
    return { status: "found", file };
  } catch (err: unknown) {
    const code = (err as NodeJS.ErrnoException | undefined)?.code;
    if (code === "ENOENT") return { status: "missing" };
    return { status: "error" };
  }
}

/**
 * Convenience reader — always returns an array. Never undefined, never throws.
 * The canonical boundary: ALL internal consumers call this, never readFileSync.
 */
export function readSessionHistory(cwd: string, goalId: string): SessionHistoryEntry[] {
  const result = readSessionHistoryDetailed(cwd, goalId);
  if (result.status === "found") return result.file.sessions;
  return [];  // missing or error → empty (fail-open, matches lease semantics)
}

function isValidSessionHistoryShape(parsed: unknown): parsed is SessionHistoryFile {
  if (typeof parsed !== "object" || parsed === null) return false;
  const p = parsed as Record<string, unknown>;
  return (
    typeof p.goalId === "string" &&
    (p.version === 1 || p.version === undefined) &&  // tolerate legacy (pre-version)
    (Array.isArray(p.sessions) || p.sessions === undefined)  // tolerate missing
  );
}
```

### Boundary contract

| Layer | Sees `sessions` as | Notes |
|---|---|---|
| `readSessionHistoryDetailed` | `SessionHistoryFile \| missing \| error` | low-level, discriminated |
| `readSessionHistory` (convenience) | `SessionHistoryEntry[]` (always) | **the boundary** |
| All other internal code | `SessionHistoryEntry[]` (always) | never `undefined`, never `null` |

### Why not make `sessions` required in the type?

Making `sessions: SessionHistoryEntry[]` required in the interface (not optional) would *look* safe but still require a runtime check — TS types don't enforce disk content. The normalization-at-read-boundary pattern is what actually protects. The required type is a **secondary** signal; the read function is the **primary** defense.

Recommendation: type `sessions` as **required** (`SessionHistoryEntry[]`) in the interface, AND normalize at read. Belt + suspenders. Readers who somehow bypass `readSessionHistory` and hit the raw file still get the type-level nudge.
